## What WinDesk Cloud is

WinDesk Cloud delivers Windows desktops as a managed cloud service. Each desktop is a dedicated virtual machine running in a regional data center, accessible over an encrypted session from any browser.

The platform has two distinct layers that work together:

- **Cloud VM layer** — dedicated virtual machines with configurable CPU, RAM, and storage resources. Each desktop runs a full Windows environment and is isolated from other tenants at the hypervisor level.
- **NeoSC zero-trust network overlay** — a cryptographic network fabric that brokers every connection between users and their desktops. There is no centralized gateway or VPN concentrator. Instead, NeoSC establishes direct peer-to-peer tunnels between authenticated endpoints, with the Control Plane coordinating identity and access policy without sitting in the data path.

This architecture eliminates the bottlenecks and single points of failure that affect traditional remote desktop and VPN infrastructure. Users connect directly to their desktop with low latency, and the network path never routes through a shared gateway that could become a target or a chokepoint.

## How it works

```text
User browser  →  NeoSC identity verification  →  Encrypted P2P tunnel  →  Windows VM
```

1. A user opens their browser and navigates to their organization's WinDesk Cloud portal.
2. NeoSC verifies the user's cryptographic identity and evaluates access policy in real time.
3. If the request is authorized, NeoSC establishes a direct encrypted tunnel to the user's assigned desktop VM.
4. The Windows desktop renders in the browser over an optimized HTML5 session — no plugins or client software required.
5. Every action during the session is logged for audit and compliance purposes.

## Architecture comparison

NeoSC's zero-trust overlay replaces legacy VPN infrastructure with a fundamentally different network model:

| | Legacy infrastructure | WinDesk Cloud (NeoSC) |
|---|---|---|
| **Network path** | All traffic routed through a central gateway | Direct peer-to-peer encrypted tunnels |
| **Authentication** | Gateway credentials, shared secrets | Cryptographic identity per device and user |
| **Failure modes** | Central gateway is a single point of failure | No centralized bottleneck; auto-failover |
| **Configuration** | Manual firewall rules, complex routing | Policy-driven, managed from the control plane |
| **Latency** | Higher — traffic detours through gateway | Lower — direct path to the desktop VM |
| **Scalability** | Gateway capacity limits throughput | Scales with the number of endpoints |
